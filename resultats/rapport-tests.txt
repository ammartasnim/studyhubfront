1. RÉSUMÉ DES TRAVAUX EFFECTUÉS

État initial :
 - 10 fichiers de test existants (auth, admin, client)
 - 1 test problématique : "Change password in settings" (échec
   intermittent dû à une assertion incorrecte)

Travaux réalisés :
 a) Correction du test "Change password in settings"
    - Ajout d'un wait explicite (toBeEnabled) avant clic
    - Remplacement de l'assertion finale par toHaveValue('')
      (vérification que le formulaire est réinitialisé)
    - Cause : le bouton reste désactivé après reset du formulaire

 b) Ajout de 10 nouveaux fichiers de test
 c) Enrichissement de 3 fichiers existants (PFP upload,
    comment/reply, send message)
 d) Configuration du répertoire de sortie "resultats/"

═══ Corrections apportées dans ce cycle ═══

 e) Admin reports — getByText('Approved') → getByRole('button')
    résolvait 2 éléments (bouton + texte) → strict mode violation
 f) Feed reply — mock GET comments utilisait items/totalItems
    (format facade) au lieu de content/totalElements (format
    Spring brut attendu par l'intercepteur HTTP)
 g) Chat send — sendMessage() nécessite WebSocket/STOMP.
    Solution : injecter le message directement dans le signal
    messagesByFriend via ng.getComponent()
 h) Feed reply — getByPlaceholder('Write a reply...') résolvait
    2 éléments (custom app-mention-input + textarea) →
    utiliser page.locator('textarea[placeholder="Write a reply..."]')
